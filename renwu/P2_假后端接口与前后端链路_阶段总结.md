# P2 假后端接口与前后端链路 阶段总结

> 本文档是 P2 阶段的学习笔记和面试讲解稿。它基于当前项目真实代码编写，重点解释“前端如何请求 Next.js 后端接口”，不提前讲 P3 真实模型、不提前讲 P7 RAG。

## 一、本阶段完成了什么

本阶段只完成“假后端接口 + 前端调用接口”的链路打通。当前没有接入外部模型接口，没有读取 `LLM_API_KEY`，没有接 Prisma / MySQL，没有保存会话和消息，没有做文档上传、解析、向量化、LanceDB、RAG、SSE、登录鉴权或 Docker。

### 1. `src/app/api/chat/route.ts`

这个文件是本阶段新增的后端接口文件。

它的作用是实现 `POST /api/chat`。前端把用户问题发送到这个接口，接口返回一个模拟回答：

```txt
这是一个模拟回答：你问的是 xxx
```

为什么要创建它：P1 的聊天回复完全在前端用 `setTimeout` 模拟，无法验证前后端链路。P2 要开始让前端真的请求 Next.js 后端接口，但仍然不接真实 AI。

它和其他文件的关系：

- 接收 `src/app/chat/page.tsx` 通过 `src/lib/request.ts` 发来的 `/api/chat` 请求；组件本身不直接写底层 `fetch` 细节。
- 使用 `src/lib/types/chat.ts` 中的 `ChatRequest`、`ChatResponse`、`ChatErrorResponse` 类型。
- 返回的数据会被前端组装成 assistant 消息，再交给 `ChatWindow` 和 `MessageBubble` 展示。

### 2. `src/lib/types/chat.ts`

这个文件是本阶段新增的聊天类型文件。

它定义了：

- `SourceChunk`
- `ChatRequest`
- `ChatResponse`
- `ChatErrorResponse`
- `ChatMessage`

为什么要创建它：P2 开始有前端和后端之间的数据交换，如果前端和后端各写一套类型，很容易字段不一致。把类型集中在 `src/lib/types/chat.ts` 后，接口和页面可以复用同一套结构。

它和其他文件的关系：

- `src/app/api/chat/route.ts` 用 `ChatRequest` 和 `ChatResponse` 约束接口输入输出。
- `src/app/chat/page.tsx` 用 `ChatRequest` 组织请求体，用 `ChatResponse` 解析响应体。
- `src/components/chat/types.ts` 从这里转发 `ChatMessage`、`SourceChunk`，让聊天组件继续使用统一类型。

### 3. `src/app/chat/page.tsx`

这个文件是本阶段修改最多的前端页面。

修改前：用户发送问题后，前端通过 `setTimeout` 本地生成模拟回复。

修改后：用户发送问题后，前端会调用：

```ts
request<ChatResponse>("/api/chat", {
  method: "POST",
  body: { question },
})
```

请求成功后，把后端返回的 `answer` 追加成 assistant 消息。

为什么要修改它：P2 的核心目标就是“前端输入 -> 后端接口 -> 后端返回 -> 前端展示”。所以聊天页不能继续纯前端自娱自乐，要真正发请求。

它和其他文件的关系：

- 调用 `ChatInput` 获取用户输入和发送动作。
- 调用 `ChatWindow` 展示消息、loading 和 error。
- 通过 `src/lib/request.ts` 发起普通 JSON 请求。
- 请求 `src/app/api/chat/route.ts`。
- 使用 `src/lib/types/chat.ts` 中的类型。

### 4. `src/lib/request.ts`

这个文件是本阶段补充新增的普通 JSON 请求封装。

它提供：

```ts
request<T>(url: string, options?: RequestOptions): Promise<T>
```

它统一处理：

- `method`，默认是 `GET`
- `headers`，默认带 `Content-Type: application/json`
- `body`，自动执行 `JSON.stringify`
- 非 2xx 状态码，统一 `throw Error`
- 成功响应，自动 `res.json()`
- 使用 TypeScript 泛型指定返回类型

为什么要创建它：P2 开始出现前端请求后端接口。如果每个组件都直接写 `fetch`，请求头、错误处理、JSON 序列化会散落在各处。抽成 `request` 后，组件只关心业务数据。

它和其他文件的关系：

- `src/app/chat/page.tsx` 使用 `request<ChatResponse>` 调用 `/api/chat`。
- 后续普通 JSON 接口也可以复用它。
- P8 的流式接口不强行放进这里，因为流式输出需要直接处理 `ReadableStream`。

### 5. `src/components/chat/ChatWindow.tsx`

这个文件是消息展示区组件。

本阶段新增了 `error?: string` 属性，并在有错误时展示红色错误提示。

为什么要修改它：一旦前端开始请求后端，就可能失败。比如接口报错、网络断开、请求体错误。如果没有错误展示，用户只会看到页面没反应。

它和其他文件的关系：

- 从 `/chat/page.tsx` 接收 `messages`、`isLoading`、`error`。
- 用 `Loading` 展示请求中状态。
- 用错误提示展示请求失败状态。

### 6. `src/components/chat/MessageBubble.tsx`

这个文件是单条消息组件。

本阶段把引用来源展示从 P1 的 `SourcePreview` 结构调整为 `SourceChunk` 结构。现在读取：

- `source.fileName`
- `source.content`
- `source.score`

为什么要修改它：P2 要提前对齐后续 RAG sources 的结构。虽然本阶段 `sources` 还是空数组，但类型上已经变成更接近真实 RAG chunk 的结构。

它和其他文件的关系：

- 接收 `ChatMessage`。
- 当 assistant 消息带有 `sources` 时，在消息下方展示“参考来源”内嵌卡片。
- 当前 P2 后端返回 `sources: []`，所以不会显示真实来源；P7 会把这里用起来。

### 7. `src/components/chat/types.ts`

这个文件原本定义聊天组件本地类型。

本阶段改成从 `src/lib/types/chat.ts` 转发：

```ts
export type { ChatMessage, SourceChunk } from "@/lib/types/chat";
```

为什么要修改它：避免前端组件类型和接口类型重复定义。组件继续从原路径引用类型，降低改动范围；真实类型来源统一迁移到 `src/lib/types/chat.ts`。

它和其他文件的关系：

- `ChatWindow`、`MessageBubble` 继续通过这个文件拿到聊天消息类型。
- `ConversationList` 保留本地 `Conversation` 类型，后续多会话阶段可继续改造。

### 8. `src/components/chat/ChatInput.tsx`

这个文件是底部输入框组件。

本阶段只修改了提示文案，从“P1 不请求后端”改为“P2 会请求后端假接口获取模拟回答”。

为什么要修改它：页面文案必须和真实行为一致。现在点击发送确实会请求 `/api/chat`，不能继续提示“不发送后端请求”。

它和其他文件的关系：

## 二、本阶段的完整开发思路

### 第一步：先定义前后端共用类型

先创建 `src/lib/types/chat.ts`，定义 `ChatRequest`、`ChatResponse`、`ChatMessage` 和 `SourceChunk`。

为什么先做这个：P2 的核心是前后端通信。通信之前必须先约定数据长什么样，否则前端以为后端返回 `answer`，后端却返回 `text`，页面就会出错。

这一步解决的问题：让请求体、响应体、消息结构有统一标准。

### 第二步：创建 `/api/chat` 假接口

创建 `src/app/api/chat/route.ts`，只导出 `POST` 方法。

接口逻辑：

1. `await req.json()` 读取请求体。
2. 从请求体里取出 `question`。
3. 如果为空，返回 400。
4. 如果正常，返回模拟 answer 和空 sources。
5. 用 try/catch 捕获异常。

为什么这样做：先做假接口，可以验证 Next.js 后端能力和前端请求流程，而不用引入真实模型 API 的复杂度。

### 第三步：封装普通 JSON 请求方法

创建 `src/lib/request.ts`，把普通接口请求的公共逻辑集中起来。

为什么这一步放在改造聊天页之前：聊天页马上要请求 `/api/chat`。如果直接在组件里写原生 `fetch`，后续每个接口都会重复写 `method`、`headers`、`JSON.stringify`、`response.ok`、`response.json()`。先封装 `request`，聊天页就能用更干净的业务写法。

这一步解决的问题：让普通 JSON 接口有统一入口。后续如果换 axios、加登录 token、加统一错误提示，只需要改 `src/lib/request.ts`。

### 第四步：改造聊天页发送逻辑

修改 `src/app/chat/page.tsx` 的 `handleSend`。

P1 的逻辑是：

```txt
点击发送 -> setTimeout -> 前端生成回复
```

P2 的逻辑变成：

```txt
点击发送 -> request("/api/chat") -> 后端返回 answer -> 前端展示 assistant 消息
```

这一步解决的问题：真正打通浏览器到 Next.js 后端的链路。

### 第五步：增加 error 状态

在 `/chat/page.tsx` 增加：

```ts
const [error, setError] = useState("");
```

并传给 `ChatWindow` 展示。

为什么要做这个：一旦开始请求后端，就必须考虑失败。没有 error 状态，用户不知道发生了什么，页面也可能一直停留在 loading。

### 第六步：保留 loading 状态

发送请求前 `setIsLoading(true)`，请求结束后在 `finally` 里 `setIsLoading(false)`。

为什么这样做：不管成功还是失败，都必须结束 loading。放在 `finally` 里可以保证请求失败时页面也不会卡住。

### 第七步：保留 sources 字段但暂时返回空数组

P2 后端返回：

```ts
sources: []
```

为什么现在为空：P2 不做 RAG 检索，还没有真实文档片段。

为什么仍然保留：P7 会返回真实引用来源，如果现在就保留字段，前端消息结构和 UI 展示位置后续不用重构。

### 和 P3、P7 的关系

P3 会把 `/api/chat` 中的模拟 answer 替换成外部 Chat API 返回的真实回答。

P7 会在 `/api/chat` 中加入 RAG 优先策略，返回真实 `sources`、回答模式和检索状态。P2 现在保留 `sources`，就是为 P7 铺路。

## 三、逐个知识点讲解

### 1. Next.js Route Handler 是什么

Route Handler 是 Next.js App Router 里的后端接口写法。

本项目位置：

```txt
src/app/api/chat/route.ts
```

只要在 `app` 目录下创建 `route.ts`，并导出 HTTP 方法函数，比如 `POST`，Next.js 就会把它当成后端接口。

为什么这里要用它：我们需要在同一个 Next.js 项目里既写前端页面，又写后端接口。Route Handler 正好适合做这种全栈项目。

如果不用会有什么问题：前端只能本地模拟，无法验证浏览器请求后端的链路。

### 2. `app/api/chat/route.ts` 为什么就是后端接口

在 App Router 中，文件路径决定路由。

`src/app/api/chat/route.ts` 对应的 URL 是：

```txt
/api/chat
```

文件里导出：

```ts
export async function POST(req: Request) {}
```

就表示它可以处理 `POST /api/chat` 请求。

它和 React 组件不同，不返回 JSX，而是返回 JSON 响应。

### 3. POST 请求是什么

POST 是 HTTP 请求方法之一，通常用于提交数据。

本项目里，用户问题不是写在 URL 上，而是放在请求体里：

```json
{
  "question": "什么是 LangChain？"
}
```

为什么用 POST：聊天提问属于提交内容，而且后续可能包含历史消息、配置、上下文等复杂数据，用 POST 更合适。

如果用 GET，会把问题拼在 URL 上，不适合长文本，也不适合后续扩展。

### 4. `req.json()` 是什么

`req.json()` 是从请求体中读取 JSON 数据的方法。

本项目代码：

```ts
const body = (await req.json()) as Partial<ChatRequest>;
```

它会把前端发送的 JSON 字符串解析成 JavaScript 对象。

为什么要 `await`：读取请求体是异步操作，需要等待完成。

如果不 `await`，拿到的是 Promise，不是实际数据。

### 5. `Response.json()` 是什么

`Response.json()` 用来返回 JSON 响应。

本项目成功返回：

```ts
return Response.json(response);
```

错误返回：

```ts
return Response.json(body, { status });
```

为什么这样用：前端 `fetch` 收到响应后，可以继续 `await response.json()` 解析。

### 6. 前端 `request` 怎么调用后端接口

本项目位置：`src/app/chat/page.tsx`

```ts
const data = await request<ChatResponse>("/api/chat", {
  method: "POST",
  body: requestBody,
});
```

关键点：

- `"/api/chat"` 是接口地址。
- `method: "POST"` 表示提交数据。
- `body: requestBody` 传普通对象即可。
- `request<ChatResponse>` 表示这个请求成功后返回 `ChatResponse` 类型。

真正的 `Content-Type`、`JSON.stringify`、错误处理和 `response.json()` 都在 `src/lib/request.ts` 里统一完成。

### 7. async / await 是什么

`async / await` 是 JavaScript 处理异步操作的写法。

本项目里：

```ts
async function handleSend() {
  const data = await request<ChatResponse>("/api/chat", ...);
}
```

请求后端需要时间，`await` 表示等请求回来再继续执行。

如果不用 `await`，代码会在请求还没完成时继续往下走，拿不到后端返回的 answer。

### 13. 为什么不在组件里散落 `fetch`

组件的职责应该是处理业务交互，比如“用户点击发送后追加消息”。原生 `fetch` 需要写很多请求细节：

- method
- headers
- body 序列化
- 判断 `response.ok`
- 解析错误响应
- 解析成功响应

如果这些都写在组件里，组件会越来越像“网络请求工具”，而不是 UI 业务组件。

本项目把这些细节放到 `src/lib/request.ts`，聊天页只写：

```ts
const data = await request<ChatResponse>("/api/chat", {
  method: "POST",
  body: requestBody,
});
```

这样组件更干净，后续也更容易维护。

### 14. `lib/request.ts` 的作用

`src/lib/request.ts` 是普通 JSON 接口的统一请求入口。

它现在底层仍然使用原生 `fetch`，没有安装 axios。它只处理普通 JSON 请求，不处理后续 P8 的流式输出。

如果后续要切换 axios，只需要保持 `request<T>()` 的调用方式不变，把 `src/lib/request.ts` 内部实现改掉即可。页面组件不需要知道底层到底是 fetch 还是 axios。

### 8. loading 状态怎么设计

本项目用：

```ts
const [isLoading, setIsLoading] = useState(false);
```

发送前：

```ts
setIsLoading(true);
```

请求结束：

```ts
finally {
  setIsLoading(false);
}
```

为什么放在 `finally`：成功或失败都会执行，避免页面一直 loading。

loading 的作用：

- 显示 `Loading` 组件。
- 禁用发送按钮，防止重复提交。

### 9. error 状态怎么设计

本项目用：

```ts
const [error, setError] = useState("");
```

请求前清空错误：

```ts
setError("");
```

请求失败时设置错误：

```ts
setError(message);
```

`ChatWindow` 接收 `error` 后展示红色提示。

为什么要这样做：接口失败时不能让用户猜发生了什么，需要明确提示。

### 10. TypeScript 请求类型和响应类型有什么作用

本项目定义：

```ts
export type ChatRequest = {
  question: string;
};

export type ChatResponse = {
  answer: string;
  sources: SourceChunk[];
};
```

作用：

- 前端发送请求时知道必须传 `question`。
- 后端返回响应时知道必须有 `answer` 和 `sources`。
- 后续修改字段时更容易发现哪里需要同步改。

如果不用类型，前端和后端字段不一致时，可能运行时才发现页面没有数据。

### 11. 为什么要保留 `sources` 字段

P2 的 `sources` 现在是空数组：

```ts
sources: []
```

但它很重要，因为 RAG 项目后续需要展示引用来源。P7 会把真实检索到的文档片段放到这里。

现在保留它，可以让前端消息结构提前适配后续 RAG。

### 12. 为什么 API Key 不能放前端

前端代码会被浏览器下载，用户可以通过开发者工具看到前端代码和公开环境变量。

如果把模型 API Key 放前端，别人就能拿到你的密钥并调用你的模型服务。

正确方式是：前端请求自己的 `/api/chat`，由 Next.js 后端在后续 P3 读取服务端环境变量并调用外部模型。P2 还没有读取任何 API Key。

## 四、关键代码逐段讲解

### 1. 后端接口读取请求体

位置：`src/app/api/chat/route.ts`

```ts
export async function POST(req: Request) {
  try {
    const body = (await req.json()) as Partial<ChatRequest>;
    const question = typeof body.question === "string" ? body.question.trim() : "";
```

这段代码解决的问题：从前端 POST 请求中读取 `question`。

关键变量：

- `req`：浏览器发来的请求对象
- `body`：解析后的请求体
- `question`：清理空格后的用户问题

数据流动：

前端组件调用 `request<ChatResponse>("/api/chat", { method: "POST", body: { question } })`。`request` 内部再使用原生 `fetch` 把 `{ question }` 发到 `/api/chat`，后端通过 `await req.json()` 解析，然后取出 `body.question`。

初学者容易看不懂的点：`req.json()` 只能读一次请求体，而且必须 `await`。

### 2. 后端空输入校验

位置：`src/app/api/chat/route.ts`

```ts
if (!question) {
  return jsonError("问题不能为空", 400);
}
```

这段代码解决的问题：防止空问题进入后续处理。

关键变量：

- `question`：如果是空字符串，说明用户没传有效问题。
- `400`：表示客户端请求不合法。

数据流动：如果前端绕过按钮禁用，直接请求空问题，后端仍然会拦住。

初学者容易忽略的点：前端校验不够，后端也必须校验。

### 3. 后端返回模拟回答

位置：`src/app/api/chat/route.ts`

```ts
const response: ChatResponse = {
  answer: `这是一个模拟回答：你问的是 ${question}`,
  sources: [],
};

return Response.json(response);
```

这段代码解决的问题：返回一个符合 `ChatResponse` 类型的模拟响应。

关键变量：

- `response.answer`：前端要展示的助手消息内容
- `response.sources`：引用来源，P2 暂时为空数组

数据流动：这个 JSON 会被前端 `await response.json()` 解析，然后变成 assistant 消息。

初学者容易看不懂的点：`Response.json` 是后端返回响应，不是前端解析响应。

### 4. 通用 request 封装

位置：`src/lib/request.ts`

```ts
export type RequestOptions = Omit<RequestInit, "body"> & {
  body?: unknown;
};

export async function request<T>(
  url: string,
  options: RequestOptions = {},
): Promise<T> {
  const { body, headers, method = "GET", ...restOptions } = options;

  const response = await fetch(url, {
    ...restOptions,
    method,
    headers: {
      "Content-Type": "application/json",
      ...headers,
    },
    body: body === undefined ? undefined : JSON.stringify(body),
  });

  if (!response.ok) {
    throw new Error(await readErrorMessage(response));
  }

  return (await response.json()) as T;
}
```

这段代码解决的问题：把普通 JSON 请求的重复细节统一封装起来。

关键变量：

- `T`：泛型，表示调用方期望拿到的响应类型
- `url`：请求地址
- `options`：请求配置
- `method`：请求方法，默认 `GET`
- `body`：请求体，可以直接传对象
- `headers`：请求头，会默认包含 JSON 类型
- `response`：原生 fetch 返回的响应对象

数据流动：

组件调用 `request<ChatResponse>`，`request` 内部把 body 转成 JSON，发出 fetch，请求成功后解析 JSON 并返回给组件。

初学者容易看不懂的点：`request<T>` 里的 `T` 不是具体变量，而是类型参数。调用时写 `request<ChatResponse>`，意思是告诉 TypeScript：这个请求成功后应该返回 `ChatResponse`。

### 5. 前端发送聊天请求

位置：`src/app/chat/page.tsx`

```ts
const requestBody: ChatRequest = { question };
const data = await request<ChatResponse>("/api/chat", {
  method: "POST",
  body: requestBody,
});
```

这段代码解决的问题：把用户问题从浏览器发送到 Next.js 后端。

关键变量：

- `requestBody`：请求体对象
- `data`：后端返回并已经解析好的 `ChatResponse`
- `method`：指定 POST
- `body`：传给后端的问题对象

数据流动：

`question` 来自输入框，封装成 `requestBody`，通过 `request` 发给 `/api/chat`。

初学者容易出错的点：容易把 `requestBody` 和最终 JSON 字符串混淆。现在组件里传对象即可，序列化由 `request` 统一做。

### 6. 前端处理错误响应

位置：`src/lib/request.ts` 和 `src/app/chat/page.tsx`

```ts
if (!response.ok) {
  throw new Error(await readErrorMessage(response));
}
```

这段代码解决的问题：后端返回 400 或 500 时，前端不能当成功处理。

关键变量：

- `response.ok`：HTTP 状态是否是成功范围
- `readErrorMessage(response)`：读取后端错误信息
- `throw new Error(...)`：进入 catch 分支

数据流动：后端 `Response.json({ error }, { status: 400 })`，前端解析 error 并展示。

初学者容易看不懂的点：fetch 遇到 400 不会自动进入 catch，必须自己判断 `response.ok`。

### 7. 前端追加助手消息

位置：`src/app/chat/page.tsx`

```ts
const data = await request<ChatResponse>("/api/chat", ...);
const assistantMessage: ChatMessage = {
  id: createMessageId("assistant"),
  role: "assistant",
  content: data.answer,
  createdAt: createTimestamp(),
  sources: data.sources,
};

setMessages((currentMessages) => [
  ...currentMessages,
  assistantMessage,
]);
```

这段代码解决的问题：把后端返回的 answer 展示到页面上。

关键变量：

- `data`：后端返回的 `ChatResponse`
- `assistantMessage`：前端消息列表中的助手消息
- `data.sources`：后端返回的引用来源，P2 是空数组

数据流动：

后端 JSON -> `data` -> `assistantMessage` -> `messages` -> `ChatWindow` -> `MessageBubble`。

初学者容易看不懂的点：接口返回的数据不能直接“自动显示”，必须转成前端消息对象并更新 state。

### 8. loading 和 error 收尾

位置：`src/app/chat/page.tsx`

```ts
} catch (requestError) {
  const message =
    requestError instanceof Error ? requestError.message : "聊天请求失败";
  setError(message);
} finally {
  setIsLoading(false);
}
```

这段代码解决的问题：请求失败时展示错误，并确保 loading 结束。

关键变量：

- `requestError`：捕获到的错误
- `message`：要展示给用户的错误文案
- `finally`：无论成功失败都会执行

初学者容易出错的点：只在成功分支关闭 loading，失败时页面会一直卡住。

## 五、本阶段的重难点

### 1. 前端为什么不能继续用假数据，而要开始请求后端

P1 的本地模拟只能验证 UI，不能验证真实网络链路。P2 需要验证：

- 前端能不能发请求
- 后端能不能读请求体
- 后端能不能返回 JSON
- 前端能不能解析响应
- loading / error 能不能正常变化

这是真实全栈项目必须跨过的一步。

### 2. 为什么要先做假接口，而不是直接接真实 AI

真实 AI 会引入很多额外问题：

- API Key 安全
- 模型服务地址
- 请求格式
- 额度和费用
- 网络失败
- 模型响应格式

如果一开始就接真实 AI，很难判断问题出在前端、后端、网络还是模型服务。假接口先把链路跑通，P3 再替换内部实现。

### 3. 为什么要处理 loading 和 error

请求后端是异步的，用户点击发送后不会立刻得到结果。

loading 解决“用户知道系统正在处理”的问题。

error 解决“请求失败时用户知道发生了什么”的问题。

没有这两个状态，用户体验会很差，也不利于调试。

### 4. 为什么要做空输入校验

空问题没有业务意义。如果不校验，会产生空消息，也会浪费接口请求。

本项目做了两层校验：

- 前端：`if (!question || isLoading) return`
- 后端：`if (!question) return 400`

前端校验提升体验，后端校验保证安全和正确性。

### 5. 为什么要统一 `ChatRequest` / `ChatResponse` 类型

前后端通信最怕结构不一致。

如果前端发送 `{ text: "..." }`，后端读取 `question`，就会拿不到数据。

如果后端返回 `{ message: "..." }`，前端读取 `answer`，页面就会显示空。

统一类型可以减少这种错误。

### 6. 为什么 sources 现在为空，但类型里要提前保留

因为 P2 不做 RAG，没有真实来源，所以返回空数组是正确的。

但 RAG 项目最终一定要展示来源。提前保留 `sources` 字段，可以让前端消息结构、组件 props 和 UI 位置提前稳定下来。P7 只需要把空数组换成真实检索结果。

### 7. 为什么要封装普通请求方法

如果组件里到处直接写 `fetch`，每个接口都要重复处理 headers、JSON.stringify、response.ok、错误解析和 response.json。重复越多，越容易出现某个页面漏写请求头、某个页面错误提示不一致的问题。

当前 `src/lib/request.ts` 把普通 JSON 请求集中起来，组件只关心：

```ts
const data = await request<ChatResponse>("/api/chat", {
  method: "POST",
  body: requestBody,
});
```

后续如果要换 axios、加 token、加统一错误拦截，都可以在 `request.ts` 里改，不需要到每个组件里找 fetch。

## 六、本阶段容易出错的地方

### 1. 请求路径写错

错误写法：

```ts
fetch("api/chat")
```

推荐写法：

```ts
fetch("/api/chat")
```

开头的 `/` 表示从网站根路径请求。

### 2. 直接写 fetch 时忘记 `JSON.stringify`

错误写法：

```ts
body: requestBody
```

正确写法：

```ts
body: JSON.stringify(requestBody)
```

HTTP 请求体需要字符串或二进制数据，不能直接传普通对象。

当前项目已经用 `request` 封装了这一步，所以组件里直接写：

```ts
body: requestBody
```

### 3. 没有设置 `Content-Type: application/json`

如果不设置，后端可能不知道请求体格式。当前代码设置了：

```ts
headers: {
  "Content-Type": "application/json",
}
```

### 4. 后端没有 `await req.json()`

错误写法：

```ts
const body = req.json();
```

这样拿到的是 Promise，不是实际对象。

正确写法：

```ts
const body = await req.json();
```

### 5. 空输入也能发送

避免方式：

```ts
const question = input.trim();
if (!question || isLoading) {
  return;
}
```

后端也要再次判断。

### 6. loading 状态结束时机不对

错误表现：请求失败后 loading 一直显示，按钮一直不能点。

避免方式：把 `setIsLoading(false)` 放到 `finally` 中。

### 7. 请求失败后页面卡住

错误表现：catch 里只打印日志，没有给用户提示。

避免方式：使用 `error` state，并传给 `ChatWindow` 展示。

### 8. 前端类型和后端返回结构不一致

错误表现：后端返回 `answer`，前端读取 `data.message`，页面没有回复。

避免方式：统一使用 `ChatRequest` 和 `ChatResponse`。

## 九、面试官可能追问的问题

### 1. Next.js 里怎么写后端接口？

可以回答：在 App Router 里创建 `app/api/.../route.ts`，然后导出 HTTP 方法函数，比如 `export async function POST(req: Request)`。本项目的接口是 `src/app/api/chat/route.ts`，对应 `/api/chat`。

### 2. Route Handler 和普通 React 组件有什么区别？

可以回答：React 组件返回 JSX，用来渲染页面；Route Handler 运行在服务端，处理 HTTP 请求，返回 JSON、文本或其他响应。`route.ts` 里不能当成页面组件写 UI。

### 3. fetch 请求为什么要写 method 和 headers？

可以回答：默认 fetch 是 GET，但我们要提交问题，所以要写 `method: "POST"`。headers 里的 `Content-Type: application/json` 是告诉后端请求体是 JSON，后端才能用 `req.json()` 解析。

### 3.1 为什么又封装了 `request`？

可以回答：因为直接在组件里写 fetch 会重复很多请求细节，比如 headers、JSON.stringify、response.ok 和错误解析。我把这些放到 `src/lib/request.ts`，组件只传 url、method 和 body。这样后续换 axios、加 token 或统一错误处理，只改一个文件。

### 4. 为什么要用 POST 而不是 GET？

可以回答：聊天问题属于提交数据，而且后续可能包含长文本、历史消息、模型参数等复杂内容。POST 更适合放在请求体里，GET 更适合读取简单资源。

### 5. loading 状态怎么设计？

可以回答：发送请求前设置 `setIsLoading(true)`，请求结束后在 `finally` 中设置 `setIsLoading(false)`。这样成功和失败都会关闭 loading，同时可以用 `isLoading` 禁用发送按钮，避免重复提交。

### 6. 前后端数据类型不一致会有什么问题？

可以回答：如果前端发送的字段和后端读取的不一致，后端拿不到问题；如果后端返回字段和前端读取的不一致，页面展示不了回答。所以我用 `ChatRequest`、`ChatResponse` 统一约束结构。

### 7. 为什么要先做 mock 接口？

可以回答：mock 接口可以先验证前端和后端链路，排除页面、请求、响应、状态管理问题。等链路稳定后，再在 P3 把模拟回答替换成真实模型调用，风险更小。

### 8. 后面接真实 AI 时，这个接口怎么改？

可以回答：P3 不需要改前端请求地址，仍然请求 `/api/chat`。主要改 `route.ts` 内部逻辑：从环境变量读取模型配置，调用外部 Chat API，把模型返回内容填到 `answer` 字段里。

### 9. 为什么 sources 现在为空？

可以回答：P2 还没有做 RAG 检索，所以没有真实来源。但保留 `sources` 字段，是为了 P7 能直接返回检索到的文档片段，前端结构不用大改。

### 10. fetch 遇到 400 会自动进 catch 吗？

可以回答：不会。fetch 只有网络错误通常才进 catch。HTTP 400/500 仍然会返回 response，所以需要自己判断 `response.ok`，如果不是 ok，再 throw Error 进入 catch。

## 十、下一阶段要做什么

下一阶段应该进入 P3：接入外部 Chat API。


改造成真实调用外部在线模型 API 后返回真实回答。

- 前端 `request<ChatResponse>("/api/chat")` 的调用方式保持不变。
