# P9 多轮对话上下文 阶段总结

本阶段只完成 **P9：多轮对话上下文**。目标是让同一个 `conversationId` 下的连续追问能够被模型理解，同时保留 P7 的 RAG 优先策略和 P8 的 SSE 流式输出能力。没有实现 P10 LangGraph、问题改写、摘要压缩、登录鉴权或 Docker。

## 一、本阶段完成了什么

### 1. `.env.example`

这个文件新增了：

```env
# Conversation
CONVERSATION_HISTORY_LIMIT=8
```

它的作用是告诉开发者：多轮对话每次最多读取最近 8 条历史消息。为什么要加这个配置？因为不能把一个会话里的所有历史都塞进 prompt，历史太多会导致 token 成本变高，也容易让很久以前的话题干扰当前问题。

它和 `src/lib/config.ts` 有关系：`.env.example` 是示例，`config.ts` 才是真正读取环境变量的地方。

### 2. `src/lib/config.ts`

这个文件新增了：

```ts
conversation: {
  historyLimit: numberEnv("CONVERSATION_HISTORY_LIMIT", 8),
}
```

它的作用是统一管理多轮对话历史条数。为什么要改这里？因为配置集中在 `config.ts` 里，后续如果想把 8 条改成 6 条或 10 条，只需要改环境变量，不用到业务代码里找数字。

它和 `src/lib/chat/history.ts` 有关系：历史查询工具会读取 `appConfig.conversation.historyLimit`。

### 3. `src/lib/chat/history.ts`

这是本阶段新增的核心工具文件，负责从 MySQL 读取当前会话最近 N 条历史消息。

它做了几件事：

1. 根据 `conversationId` 和 `userId` 校验会话是否属于当前用户。
2. 按 `createdAt desc` 查询最近 N 条消息。
3. 再 `reverse()` 成时间正序。
4. 只保留 `user` 和 `assistant` 消息。
5. 把数据库消息转换成大模型需要的 `ChatProviderMessage[]`。
6. 对过长消息做简单截断。

为什么要创建这个文件？因为历史消息查询会被普通接口和流式接口共同使用，不能在两个 Route Handler 里各写一套。

它和 `prepare-answer.ts` 有关系：`prepareRagPriorityAnswer` 会调用它拿到 `historyMessages`。

### 4. `src/lib/rag/build-prompt.ts`

这个文件原来只负责把 `question + sources` 拼成 RAG Prompt。本阶段修改为支持：

```ts
historyMessages?: ChatProviderMessage[];
```

现在 RAG Prompt 的结构是：

```txt
system：知识库问答规则
historyMessages：当前会话历史消息
user：当前用户问题 + 参考资料 sources
```

为什么要这样改？因为 RAG 回答不只需要知识库资料，还需要理解用户追问里的“第二点”“它”“上面那个”等上下文。历史消息负责理解对话关系，sources 负责提供知识依据。

### 5. `src/lib/rag/build-fallback-prompt.ts`

这是新增文件，负责构造普通模型兜底回答的 Prompt。

它解决的问题是：即使没有命中知识库，fallback 回答也应该理解当前会话历史。例如用户追问“能举个例子吗？”，即使走普通模型，也需要知道上一轮在讲什么。

它和 `prepare-answer.ts` 有关系：当 `retrievalStatus !== "hit"` 时，`prepareRagPriorityAnswer` 会调用它。

### 6. `src/lib/rag/prepare-answer.ts`

这是 P8 已经拆出的“RAG 准备流程”，P9 继续改造它，而不是重写一套。

现在它的入参变成：

```ts
{
  question: string;
  conversationId?: string;
  userId: string;
}
```

它的流程是：

```txt
接收 question、conversationId、userId
→ 如果有 conversationId，读取最近历史消息
→ 执行 RAG 优先检索
→ 命中：buildRagMessages(question + history + sources)
→ 未命中：buildFallbackMessages(question + history + fallbackReason)
→ 返回 messages + answerMode + retrievalStatus + sources
```

这个文件是 P9 的中枢，因为普通接口和流式接口都复用它。

### 7. `src/lib/rag/answer-question.ts`

这个文件保留了一次性回答的封装，但现在也会传入 `conversationId` 和 `userId`。它内部调用 `prepareRagPriorityAnswer`，再调用模型 Provider 生成完整回答。

它的作用是保持“准备 Prompt”和“调用模型”之间的职责清晰。

### 8. `src/app/api/chat/route.ts`

这是普通一次性回答接口。本阶段重点改造了它的顺序：

```txt
创建或校验 conversation
→ prepareRagPriorityAnswer 读取当前问题之前的历史消息
→ 保存当前 user 消息
→ 调用模型生成 answer
→ 保存 assistant 消息
→ 返回 ChatResponse
```

为什么先读取历史再保存当前 user 消息？因为如果先保存当前问题，再查询最近历史，当前问题会出现在历史消息里，然后又作为当前问题被放到 prompt 最后，导致重复注入。

### 9. `src/app/api/chat/stream/route.ts`

这是 P8 的 SSE 流式接口。本阶段让它也支持同一套多轮上下文。

它保留了：

```txt
meta
token
done
error
```

同时改成调用：

```ts
prepareRagPriorityAnswer({
  question,
  conversationId: conversation.id,
  userId: MOCK_USER_ID,
})
```

这样普通接口和流式接口不会出现“一个支持多轮，一个不支持多轮”的不一致。

### 10. `src/app/chat/page.tsx`

前端聊天页做了小范围优化：

1. 切换会话前先 `abort()` 当前流式请求。
2. 新建会话前也清理当前流式状态。
3. 清空 `streamingMessageIdRef`，避免旧 token 写入新会话。

为什么要改？因为多轮上下文的边界是 `conversationId`。如果 A 会话正在流式生成时切到 B 会话，旧请求继续回来就可能污染 B 会话页面。

### 11. `src/components/common/MarkdownRenderer.tsx`

本阶段顺手修复了 P8 遗留的构建错误：`react-markdown@10` 的 `code` 组件类型里没有 `inline` 字段，所以改为用 `className` 判断行内代码和代码块。

这个修改不是 P9 新功能，但它是 P9 验证必须修的构建阻塞点。

## 二、本阶段的完整开发思路

第一步，我先做历史消息查询工具。原因是多轮对话的基础不是前端状态，而是数据库里已经保存的 `Message`。P4 已经把消息落库，P9 就可以基于 `conversationId` 从数据库读取历史。

第二步，我改造 RAG Prompt。RAG 回答要同时看两类信息：历史消息和知识库 sources。历史消息用于理解追问，sources 用于提供事实依据，所以它们在 prompt 里不能混成一团。

第三步，我单独做 fallback Prompt。因为 fallback 不是“不需要上下文”，它只是“没有使用知识库来源”。普通模型回答同样需要理解历史追问。

第四步，我改造 `prepareRagPriorityAnswer`。这个函数是 P8 拆出来的公共准备层，所以 P9 把历史消息接在这里最合适。这样 `/api/chat` 和 `/api/chat/stream` 都能复用同一套多轮上下文逻辑。

第五步，我改造两个聊天接口。普通接口和流式接口都采用“先读历史、再保存当前用户消息”的顺序，解决当前问题重复进入 prompt 的问题。

第六步，我检查前端会话切换。多轮上下文必须按会话隔离，所以切换会话时要停止旧的流式请求，避免旧会话输出污染新会话。

P9 和后续 P10 的关系：P9 先把“历史消息 + RAG 检索 + fallback”这条基础多轮链路打通。P10 如果进入 LangGraph，就可以把历史消息作为 Agent 状态的一部分，继续做问题改写、检索判断和答案生成节点。

## 三、逐个知识点讲解

### 1. 什么是多轮对话

多轮对话就是模型不只看当前这一句，还能结合前面几轮问答理解上下文。比如用户说“第二点详细说一下”，模型需要知道上一轮回答里的第二点是什么。

本项目里，多轮对话通过 `src/lib/chat/history.ts` 从数据库读取历史消息实现。

### 2. 什么是 conversationId

`conversationId` 是会话 ID。一个会话下面有多条 `Message`。它是多轮对话的边界：A 会话的历史只能给 A 会话用，不能混到 B 会话。

本项目里，前端发送消息时会把当前 `conversationId` 传给 `/api/chat/stream`。

### 3. 为什么只取最近 N 条历史消息

因为大模型有上下文长度限制，历史消息越多，prompt 越长，成本越高，也更容易引入旧话题干扰。P9 默认读取最近 8 条。

对应代码在 `appConfig.conversation.historyLimit`。

### 4. historyMessages 是什么

`historyMessages` 是从数据库消息转换出来的：

```ts
ChatProviderMessage[]
```

它只包含：

```ts
role
content
```

不会把 `sources`、`answerMode`、`retrievalStatus` 这些展示元信息塞给模型。

### 5. historyMessages 和 sources 的区别

historyMessages 是当前会话里之前说过什么，用来理解上下文。sources 是向量检索命中的知识库片段，用来提供事实依据。

RAG 里二者都重要，但作用不同。

### 6. RAG 模式下历史消息和知识库资料怎么配合

RAG 命中时，prompt 会包含：

```txt
历史消息
当前问题
参考资料
```

模型可以用历史消息理解“这个”“第二点”，再用参考资料保证答案可追溯。

### 7. fallback 模式下历史消息怎么发挥作用

fallback 没有 sources，但仍然会把历史消息传给模型。这样用户继续追问时，普通模型也能理解上下文。

### 8. 为什么当前问题不能重复注入 prompt

如果先保存当前 user 消息，再读取最近 8 条历史，当前问题可能已经在历史里。然后 prompt 最后又会追加当前问题，模型会看到两遍同一个问题。

本项目选择：先读取历史，再保存当前 user 消息。

### 9. 为什么多轮上下文是后续 LangGraph 的基础

LangGraph 后续会把问题改写、检索、相关性判断、生成回答拆成节点。每个节点都需要知道当前问题和历史上下文，所以 P9 先把历史上下文整理好。

## 四、关键代码逐段讲解

### 1. 历史消息查询

位置：`src/lib/chat/history.ts`

```ts
const recentMessages = await prisma.message.findMany({
  where: {
    conversationId: params.conversationId,
    role: { in: ["user", "assistant"] },
  },
  orderBy: { createdAt: "desc" },
  take: Math.max(0, limit),
});
```

这段代码解决“从数据库取最近几条消息”的问题。`orderBy desc` 是为了先拿到最新消息，`take` 是限制条数。

初学者容易看不懂的是：拿到后还要 `reverse()`。因为数据库倒序查询更方便，但模型阅读对话需要正序。

### 2. prepare 中读取历史

位置：`src/lib/rag/prepare-answer.ts`

```ts
const historyMessages = params.conversationId
  ? await getRecentConversationMessages({
      conversationId: params.conversationId,
      userId: params.userId,
      limit: appConfig.conversation.historyLimit,
    })
  : [];
```

这段代码解决“有会话就查历史，没有会话就空历史”的问题。

数据流是：`conversationId` → MySQL `Message` → `ChatProviderMessage[]` → prompt。

### 3. RAG Prompt 加历史

位置：`src/lib/rag/build-prompt.ts`

```ts
return [
  { role: "system", content: RAG_SYSTEM_PROMPT },
  ...(params.historyMessages ?? []),
  {
    role: "user",
    content: ["当前用户问题：", params.question, "", "参考资料：", sourceText].join("\n"),
  },
];
```

这段代码解决“历史消息、当前问题、参考资料如何排序”的问题。

当前问题放最后，是为了让模型知道最终要回答哪一句。

### 4. fallback Prompt 加历史

位置：`src/lib/rag/build-fallback-prompt.ts`

```ts
return [
  { role: "system", content: systemContent },
  ...(params.historyMessages ?? []),
  { role: "user", content: params.question },
];
```

这段代码说明 fallback 也不是孤立回答，而是带着历史消息回答当前问题。

### 5. 普通接口避免重复注入

位置：`src/app/api/chat/route.ts`

```ts
const prepared = await prepareRagPriorityAnswer({
  question,
  conversationId: conversation.id,
  userId: MOCK_USER_ID,
});

await prisma.message.create({
  data: {
    conversationId: conversation.id,
    role: "user",
    content: question,
  },
});
```

这段代码的关键是顺序：先 prepare，后保存当前 user 消息。这样历史消息里不会包含当前问题。

### 6. 流式接口复用同一套 prepare

位置：`src/app/api/chat/stream/route.ts`

```ts
const prepared = await prepareRagPriorityAnswer({
  question,
  conversationId: conversation.id,
  userId: MOCK_USER_ID,
});
```

这段代码保证 `/api/chat/stream` 和 `/api/chat` 使用相同上下文逻辑。否则很容易出现普通接口支持多轮，流式接口不支持多轮的问题。

### 7. 前端切换会话时停止流式请求

位置：`src/app/chat/page.tsx`

```ts
abortControllerRef.current?.abort();
setActiveConversationId(conversationId);
setStreamStatus("idle");
setStreamError("");
streamingMessageIdRef.current = null;
```

这段代码解决“切换会话时旧流式响应还在回来”的问题。它避免 A 会话正在生成的 token 写入 B 会话页面。

## 五、本阶段的重难点

### 1. 多轮对话不是简单把所有消息拼起来

如果把全部消息都拼进去，prompt 会越来越长，成本越来越高，甚至超过模型上下文限制。P9 只取最近 N 条，是第一版最稳的方案。

### 2. 历史消息顺序很重要

模型读对话需要从旧到新。如果顺序反了，模型可能误解谁先问、谁后答。

### 3. 当前问题不能重复出现

当前问题既是本轮要回答的问题，又可能刚被保存进数据库。P9 通过“先读历史，再保存当前 user 消息”避免重复。

### 4. RAG 里的 history 和 sources 不能混在一起

history 是对话上下文，sources 是知识库依据。如果把 sources 当历史消息，模型可能把资料片段误认为用户说过的话。

### 5. 会话切换必须隔离上下文

多轮上下文必须绑定 `conversationId`。如果前端切换会话不清理状态，旧会话的生成结果可能污染新会话。

### 6. 普通接口和流式接口必须复用 prepare

P8 之后项目有两个聊天入口：一次性 `/api/chat` 和流式 `/api/chat/stream`。P9 把上下文逻辑放在 `prepareRagPriorityAnswer`，避免两边行为不一致。

### 7. P9 暂时不做摘要压缩

摘要压缩是更高级的 memory 能力，会引入额外模型调用和状态管理。P9 先用最近 N 条消息，足够验证连续追问。

## 六、本阶段容易出错的地方

1. 历史消息顺序反了：查询时可以倒序，但传给模型前必须正序。
2. 当前问题重复传给模型：要么先查历史再保存当前消息，要么查询时排除当前消息。本项目选择前者。
3. 不同会话上下文串了：历史查询必须用 `conversationId + userId`。
4. 查询历史消息时没有校验 userId：后续多用户时会有数据安全问题。
5. 把 sources 当成历史消息：sources 应该放在当前问题的参考资料里。
6. 把所有历史消息都塞进 prompt：会造成上下文过长。
7. 普通接口支持了多轮，但流式接口忘记支持：所以本项目把逻辑集中到 `prepareRagPriorityAnswer`。
8. 切换会话后前端状态没清理：需要停止旧流式请求并重新加载消息。
9. 没有 conversationId 的新会话场景：后端会自动创建会话，前端收到 meta 后更新当前会话 ID。

## 八、面试时我应该怎么说

### 30 秒简短版

在 P9 阶段，我主要实现了多轮对话上下文。我的做法是基于 `conversationId` 从数据库读取当前会话最近 8 条历史消息，把它们整理成模型需要的 messages，再拼进 RAG 或 fallback 的 Prompt 里。这样用户在同一个会话里继续追问“第二点详细说一下”时，模型能结合上一轮回答理解上下文。同时我让普通接口和 SSE 流式接口复用同一套上下文准备逻辑，避免两边行为不一致。

### 1 分钟详细版

在 P9 阶段，我没有重新做一套聊天系统，而是在 P4 已经保存会话消息、P7 已经完成 RAG 优先、P8 已经完成流式输出的基础上加入历史上下文。我先写了 `getRecentConversationMessages`，根据 `conversationId` 和 `userId` 查询最近 N 条消息，并按时间正序返回给模型。然后我改造 RAG Prompt 和 fallback Prompt，让它们都能接收 `historyMessages`。RAG 命中时，模型会同时看到历史消息、当前问题和知识库 sources；fallback 时虽然没有 sources，但也会看到历史消息。为了避免当前问题重复出现在 prompt 里，我选择先读取历史，再保存当前用户消息。最后我把 `/api/chat` 和 `/api/chat/stream` 都接到同一个 `prepareRagPriorityAnswer`，保证普通回答和流式回答都支持多轮上下文。

### 追问展开版

如果继续展开，我会说这个阶段的关键不是简单拼字符串，而是控制上下文边界和数据顺序。上下文边界通过 `conversationId` 控制，保证不同会话之间不会串。历史消息数量通过 `CONVERSATION_HISTORY_LIMIT` 控制，避免 prompt 过长。数据库查询最近消息时我用倒序取最近 N 条，再反转成正序给模型，因为模型需要按真实对话顺序理解上下文。RAG 模式下，我把历史消息和 sources 分开处理：历史消息帮助理解追问，sources 提供知识依据；fallback 模式下，也继续传历史消息，让普通模型回答也能连续。这个设计也为后续 LangGraph 做准备，因为后面可以把历史消息作为状态传给问题改写、检索和生成节点。

## 九、面试官可能追问的问题

### 1. 多轮对话是怎么实现的？

可以回答：基于 `conversationId` 查询当前会话最近 N 条 `Message`，转换成模型 messages，再和当前问题一起发给模型。

### 2. conversationId 在项目里有什么作用？

可以回答：它是会话边界。一个 `Conversation` 下有多条消息，模型只能读取当前会话的历史，不能读其他会话。

### 3. 为什么只取最近 N 条？

可以回答：为了控制 prompt 长度和 token 成本，也避免旧话题干扰当前问题。

### 4. 历史消息顺序为什么重要？

可以回答：模型需要按真实聊天顺序理解谁先问、谁后答。数据库可以倒序取最近消息，但传给模型前要转成正序。

### 5. 怎么避免上下文太长？

可以回答：第一版限制最近 8 条，并对单条过长消息做截断。后续可以做摘要压缩。

### 6. 怎么避免当前问题重复注入？

可以回答：本项目选择先读取历史消息，再保存当前 user 消息。这样当前问题只作为 prompt 最后一条 user 消息出现一次。

### 7. RAG 里的 history 和 sources 有什么区别？

可以回答：history 是对话上下文，用于理解追问；sources 是知识库检索结果，用于提供事实依据和引用来源。

### 8. 切换会话时怎么保证上下文不串？

可以回答：前端切换会话时设置新的 `activeConversationId` 并重新加载消息，同时停止旧的流式请求。后端查询历史时也用 `conversationId + userId` 校验。

### 9. 流式输出下怎么支持多轮上下文？

可以回答：流式接口在开始输出 token 前，也会先调用 `prepareRagPriorityAnswer` 拼好带历史的 messages，然后用模型的 stream 方法逐段输出。

### 10. 后续历史消息很多怎么优化？

可以回答：可以做摘要压缩、按 token 预算截断、保留最近对话加长期摘要，或者在 LangGraph 中加入 memory 节点。

### 11. P9 和 LangGraph 有什么关系？

可以回答：P9 先准备好历史上下文能力。P10/P11 做 LangGraph 时，可以把历史消息作为状态传给问题改写、检索、相关性判断和答案生成节点。

## 十、下一阶段要做什么

下一阶段应该进入 **P10：LangGraph Agentic RAG**。

当前 P9 已经具备：

```txt
历史消息
+ RAG 检索
+ fallback
+ SSE 流式输出
```

P10 会进一步把这些流程拆成更明确的状态节点，例如问题改写、是否检索、向量检索、相关性判断、RAG 生成、fallback 兜底等。这样项目会从“函数式 RAG 流程”升级成“可观察、可扩展的 Agentic RAG 流程”。
