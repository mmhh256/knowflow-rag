# P5 知识库文档上传与解析 阶段总结

## 一、本阶段完成了什么

本阶段只完成 P5：知识库文档上传、文档管理和文本解析，没有做 embedding、LanceDB、RAG、SSE 或登录。

1. `prisma/schema.prisma`
   - 作用：在 `Document` 模型中新增 `parsedText String? @db.LongText` 和 `parseError String? @db.Text`。
   - 为什么修改：P5 需要把 PDF、TXT、Markdown 解析出来的纯文本保存到数据库，解析失败时也要记录失败原因。
   - 和其他文件的关系：上传接口会写入 `Document` 表，`/documents` 页面会读取这些字段生成预览和状态展示。

2. `prisma/migrations/20260530074522_add_document_parsing_fields/migration.sql`
   - 作用：把 `parsedText` 和 `parseError` 两个字段真正同步到 MySQL。
   - 为什么创建：只改 Prisma schema 不会自动改数据库，migration 才是数据库结构变更记录。
   - 和其他文件的关系：它来自 `schema.prisma` 的变更，执行后 Prisma Client 才能读写新字段。

3. `src/lib/types/document.ts`
   - 作用：定义 `DocumentStatus`、`KnowledgeDocument`、上传响应、列表响应和详情响应类型。
   - 为什么创建：前端和后端都需要统一文档数据结构，避免字段名和状态字符串写散。
   - 和其他文件的关系：`/documents` 页面和文档 API 都复用这些类型。

4. `src/lib/documents/document-types.ts`
   - 作用：集中管理支持的文件类型、最大上传大小、状态文案、文件大小格式化和 DTO 转换。
   - 为什么创建：文档相关规则不能散落在页面和 API 里，否则后面扩展格式时很难维护。
   - 和其他文件的关系：上传 API 用它校验类型和大小，前端页面用它显示状态和文件大小。

5. `src/lib/documents/save-upload-file.ts`
   - 作用：把浏览器上传的文件保存到本地 `uploads` 目录。
   - 为什么创建：文件必须落到后端目录，后端才能解析、后续分块和向量化。
   - 和其他文件的关系：`POST /api/documents/upload` 调用它保存文件，然后把返回的 `filePath` 写入数据库。

6. `src/lib/documents/parse-document.ts`
   - 作用：统一文档解析入口，根据 `fileType` 分发到 PDF 或文本解析器。
   - 为什么创建：以后如果支持 DOCX、XLSX，只需要在这里新增分支。
   - 和其他文件的关系：上传接口只调用 `parseDocument`，不用关心具体解析细节。

7. `src/lib/documents/parse-pdf.ts`
   - 作用：使用 `pdf-parse` 读取 PDF 文本层。
   - 为什么创建：P5 第一版要支持 PDF 解析，但只解析 PDF 自带文本，不做 OCR。
   - 和其他文件的关系：由 `parse-document.ts` 调用，解析结果保存到 `Document.parsedText`。

8. `src/lib/documents/parse-text.ts`
   - 作用：读取 TXT / Markdown 文件，并做基础空白清理。
   - 为什么创建：Markdown 在 P5 先按普通文本处理，不提前做复杂语法解析。
   - 和其他文件的关系：由 `parse-document.ts` 调用，结果同样写入 `parsedText`。

9. `src/app/api/documents/upload/route.ts`
   - 作用：实现 `POST /api/documents/upload`，接收文件、校验、保存、解析、写数据库。
   - 为什么创建：文件上传必须走后端接口，前端不能直接写服务器文件系统。
   - 和其他文件的关系：调用 `saveUploadFile`、`parseDocument`、`ensureMockUser` 和 Prisma。

10. `src/app/api/documents/route.ts`
    - 作用：实现 `GET /api/documents`，读取当前 mock 用户的文档列表。
    - 为什么创建：让知识库页面展示真实数据库记录，不再用静态假数据。
    - 和其他文件的关系：返回 `KnowledgeDocument[]` 给 `/documents` 页面。

11. `src/app/api/documents/[id]/route.ts`
    - 作用：实现 `GET /api/documents/[id]` 和 `DELETE /api/documents/[id]`。
    - 为什么创建：支持查看文档详情和删除文档。
    - 和其他文件的关系：删除时同时删数据库记录和本地 `uploads` 文件。

12. `src/app/documents/page.tsx`
    - 作用：把静态知识库页面改造成真实文档管理页面。
    - 为什么修改：P5 要能选择文件、上传、查看列表、查看详情、删除文档。
    - 和其他文件的关系：通过 `request` 调用文档 API，用 `KnowledgeDocument` 类型约束页面状态。

13. `src/lib/request.ts`
    - 作用：让通用请求封装同时支持 JSON 和 FormData。
    - 为什么修改：文件上传不能 `JSON.stringify`，也不能手动设置 `Content-Type: application/json`。
    - 和其他文件的关系：`/documents` 页面上传文件时继续复用统一请求封装。

14. `package.json` 和 `package-lock.json`
    - 作用：新增 `pdf-parse` 依赖。
    - 为什么修改：P5 需要从 PDF 中提取文本层内容。

15. `.gitignore`
    - 作用：忽略 `uploads` 里的真实上传文件，但保留 `uploads/.gitkeep`。
    - 为什么修改：用户上传的本地文件不应该提交到 Git。

## 二、本阶段的完整开发思路

我先调整 `Document` 表，因为上传解析的结果最终必须有地方保存。只靠前端 `useState` 能显示一会儿，但刷新页面就没了，所以 P5 的第一步是给数据库补 `parsedText` 和 `parseError`。

然后我做文件保存工具 `save-upload-file.ts`。这个工具解决两个问题：第一，把 `FormData` 里的文件真正写入 `uploads`；第二，用 `randomUUID()` 避免同名文件互相覆盖。

接着我做解析工具。`parse-document.ts` 是统一入口，`parse-pdf.ts` 负责 PDF，`parse-text.ts` 负责 TXT 和 MD。这样上传接口不需要知道每种文件怎么解析，只要传入文件路径和文件类型。

然后我做文档 API。上传接口负责完整流程：接收文件、校验类型和大小、保存文件、创建数据库记录、更新状态为 `parsing`、解析文本、最后改成 `parsed` 或 `parse_failed`。列表接口只返回预览，不返回完整 `parsedText`，避免列表太重。详情接口返回更长预览。删除接口会同时删数据库记录和本地文件。

最后我改造 `/documents` 页面，把原来的静态占位替换成真实交互。页面现在会加载文档列表、上传文件、显示状态、查看详情、删除文档。

P5 和后续 P6、P7 的关系很直接：P5 产出 `parsedText`，P6 会把 `parsedText` 切成 chunk 并做 embedding，P7 会把检索命中的 chunk 作为 sources 返回给聊天回答。

## 三、逐个知识点讲解

**知识库页面是什么**
知识库页面就是管理资料入口。本项目里是 `src/app/documents/page.tsx`。它负责让用户上传 PDF、TXT、MD，并看到哪些文档已经解析成功。没有知识库页面，后面的 RAG 就没有文档来源。

**文件上传是什么**
文件上传就是把浏览器本地文件传给后端。本项目里通过 `<input type="file" />` 选择文件，再用 `FormData` 发送到 `/api/documents/upload`。

**FormData 是什么**
FormData 是浏览器专门用来提交文件和表单字段的数据格式。本项目里：

```ts
const formData = new FormData();
formData.append("file", selectedFile);
```

如果不用 FormData，而是普通 JSON，文件对象不会被正确传到后端。

**multipart/form-data 是什么**
这是文件上传时浏览器自动生成的请求格式。它会带一个 boundary，把文件内容和字段分开。这个 boundary 必须由浏览器生成，所以 `request.ts` 里遇到 FormData 时不会手动设置 `Content-Type`。

**Next.js Route Handler 如何接收文件**
`src/app/api/documents/upload/route.ts` 里使用：

```ts
const formData = await req.formData();
const file = formData.get("file");
```

这段代码把前端传来的文件取出来。它运行在后端，不会暴露文件系统给浏览器。

**为什么文件上传要用 Node.js Runtime**
上传接口写了：

```ts
export const runtime = "nodejs";
```

因为保存文件和 PDF 解析都需要 Node.js 的 `fs` 能力，Edge Runtime 不适合做这些文件系统操作。

**为什么文件要保存到 uploads**
`save-upload-file.ts` 会把文件写入项目根目录下的 `uploads`。这样后端后续可以读取它、解析它、删除它。只放在前端内存里，刷新页面就丢了。

**为什么要校验文件类型和大小**
P5 只支持 PDF、TXT、MD，最大 10MB。校验放在前端和后端两边：前端为了用户体验，后端为了安全。真正可靠的是后端校验。

**pdf-parse 是什么**
`pdf-parse` 是解析 PDF 文本层的库。本项目里 `parse-pdf.ts` 用 `PDFParse` 读取 PDF 内容。它不是 OCR，所以扫描件图片 PDF 可能解析不到文字。

**Markdown 为什么可以先按文本处理**
Markdown 本质上是纯文本，里面的 `#`、`-`、代码块都可以先作为文本保存。P5 的目标是得到可分块的文本，不需要提前渲染 Markdown。

**Document 表为什么需要 parsedText**
`parsedText` 是后续 P6 的输入。没有它，P6 就没有文本可以切块，也无法生成 embedding。

**文档状态 status 有什么用**
状态让前端知道文档处理进度。本阶段主要用：

- `uploaded`：已上传
- `parsing`：解析中
- `parsed`：已解析
- `parse_failed`：解析失败

后续 P6 会继续使用 `indexing`、`indexed`、`index_failed`。

## 四、关键代码逐段讲解

**1. 上传页面的 FormData**

位置：`src/app/documents/page.tsx`

```ts
const formData = new FormData();
formData.append("file", selectedFile);

const data = await request<DocumentUploadResponse>("/api/documents/upload", {
  method: "POST",
  body: formData,
});
```

这段代码解决“前端怎么把文件传给后端”的问题。`selectedFile` 是用户在文件框里选中的文件，`formData` 是上传容器，`request<DocumentUploadResponse>` 让返回结果有明确类型。

初学者容易误解的点是：文件不能像普通对象一样 `JSON.stringify`，否则后端拿不到真正的文件内容。

**2. request 对 FormData 的特殊处理**

位置：`src/lib/request.ts`

```ts
const isFormData = body instanceof FormData;

if (!isFormData && !finalHeaders.has("Content-Type")) {
  finalHeaders.set("Content-Type", "application/json");
}
```

这段代码解决“普通 JSON 请求和文件上传请求如何共用一个 request”的问题。普通对象自动走 JSON，FormData 不设置 JSON 请求头。

数据流是：页面传入 `FormData`，`request` 识别到它是文件上传，就让浏览器自己处理 multipart 请求头。

**3. 上传接口完整流程**

位置：`src/app/api/documents/upload/route.ts`

```ts
const formData = await req.formData();
const file = formData.get("file");
const savedFile = await saveUploadFile(file);

let document = await prisma.document.create({
  data: {
    userId: MOCK_USER_ID,
    fileName: savedFile.fileName,
    fileType: savedFile.fileType,
    fileSize: savedFile.fileSize,
    filePath: savedFile.filePath,
    status: "uploaded",
  },
});
```

这段代码解决“后端怎么接收文件并保存文档记录”的问题。`file` 是前端传来的文件，`savedFile` 是保存到本地后的结果，`document` 是 MySQL 中的记录。

数据先从浏览器到 `/api/documents/upload`，再写入 `uploads`，再写入 `Document` 表。

**4. 状态流转和解析**

位置：`src/app/api/documents/upload/route.ts`

```ts
document = await prisma.document.update({
  where: { id: document.id },
  data: { status: "parsing" },
});

const parsedText = await parseDocument(savedFile.filePath, savedFile.fileType);

document = await prisma.document.update({
  where: { id: document.id },
  data: {
    status: "parsed",
    parsedText,
    parseError: null,
    chunkCount: 0,
  },
});
```

这段代码解决“文档处理过程如何记录”的问题。`status` 从 `uploaded` 到 `parsing`，解析成功后变成 `parsed`。如果解析失败，会进入 `parse_failed` 并保存 `parseError`。

初学者容易不理解的是：状态不是为了数据库好看，而是为了前端和后续任务知道文档现在可不可以进入下一步。

**5. 列表接口不返回完整 parsedText**

位置：`src/app/api/documents/route.ts`

```ts
return Response.json({
  documents: documents.map(toKnowledgeDocument),
});
```

`toKnowledgeDocument` 只生成 `preview`，不把完整 `parsedText` 返回给列表页。这样可以避免一个大 PDF 让列表接口响应很慢。

## 五、本阶段的重难点

1. 文件上传不能像普通 JSON 一样处理。普通 JSON 适合传 `{ question: "你好" }`，文件必须用 FormData。
2. FormData 不要手动设置 `Content-Type`。浏览器会自动生成 multipart boundary，手动设置反而容易让后端读不到文件。
3. 文档状态必须清晰。`uploaded`、`parsing`、`parsed`、`parse_failed` 能让用户知道文档卡在哪一步。
4. 列表接口不要返回完整 `parsedText`。列表只需要展示文件信息和预览，完整文本后续可以走详情或专门接口。
5. 保存 `parsedText` 是 P6 的基础。P6 不会再从 PDF 重新解析，而是直接读取数据库里的纯文本。
6. P5 不能做 embedding。因为当前阶段只负责“把文档变成文本”，向量化是下一阶段的职责。
7. 删除文档要同时处理数据库和本地文件。只删数据库会留下无用文件，只删文件会留下脏数据。

## 六、本阶段容易出错的地方

1. `uploads` 目录不存在：`saveUploadFile` 用 `mkdir(..., { recursive: true })` 自动创建。
2. 文件名冲突：保存文件时使用 `randomUUID()`，避免同名覆盖。
3. 直接信任用户文件名：`sanitizeFileName` 会做基础安全处理。
4. FormData 请求头设置错：`request.ts` 遇到 FormData 不设置 JSON Content-Type。
5. 上传接口没有设置 Node.js runtime：已在上传接口中写 `export const runtime = "nodejs"`。
6. PDF 是扫描件：`pdf-parse` 只能读文本层，扫描件需要 OCR，P5 暂不支持。
7. 列表接口返回太重：接口只返回预览，不返回完整 parsedText。
8. 删除本地文件失败导致接口崩溃：删除文件失败会被捕获，不影响数据库删除。
9. 修改 Prisma schema 后忘记 migrate：本阶段已执行 `npm run prisma:migrate -- --name add_document_parsing_fields`。
10. `pdf-parse` 新版 worker 和 Next/Turbopack 不兼容：我上传 PDF 时遇到过 `Cannot find module ... pdf.worker.mjs`，最后把 `pdf-parse` 固定到 1.x 稳定版本，并从 `pdf-parse/lib/pdf-parse` 引入解析函数，避免新版 worker 打包问题。
11. Prisma Client 没重启导致字段不认识：我遇到过 `Unknown argument parseError`，原因是 dev server 还在用旧的 Prisma Client。解决方式是执行 `npm run prisma:generate`，并重启 `npm run dev`。

## 九、面试官可能追问的问题

1. 文件上传为什么用 FormData？
   - 因为文件是二进制内容，不能直接用普通 JSON 可靠传输。FormData 是浏览器标准的文件上传方式。

2. FormData 和 JSON 请求有什么区别？
   - JSON 适合传普通对象；FormData 可以传文件和字段，浏览器会用 multipart/form-data 组织请求。

3. 为什么不能手动设置 multipart 的 Content-Type？
   - multipart 需要 boundary，浏览器会自动生成。手动写容易缺 boundary，后端就读不到文件。

4. 上传文件为什么要在后端处理？
   - 因为只有后端能写服务器文件系统、保存数据库记录、做解析和后续向量化。

5. pdf-parse 能解析所有 PDF 吗？
   - 不能。它只能解析 PDF 文本层，扫描件图片 PDF 需要 OCR，P5 没做 OCR。

6. 为什么要保存 parsedText？
   - 后续 P6 要直接读取 parsedText 做文本分块和 embedding，不需要每次重新解析文件。

7. 为什么列表接口不直接返回完整 parsedText？
   - 完整文本可能很大，列表页只需要预览。返回完整文本会增加响应体积和页面负担。

8. 文档状态是怎么设计的？
   - P5 主要是 uploaded、parsing、parsed、parse_failed；P6 会继续使用 indexing、indexed、index_failed。

9. 删除文档时为什么要删文件和数据库？
   - 数据库记录和本地文件是一组资源，只删其中一个都会留下脏数据。

10. P5 和后面的 embedding 有什么关系？
    - P5 产出 parsedText，P6 会把 parsedText 切成 chunk，再调用 embedding API 生成向量。

## 十、下一阶段要做什么

下一阶段应该进入 **P6：文本分块与 Embedding 向量化**。

P6 会继续使用 P5 保存下来的 `Document.parsedText`。具体流程会是：读取已解析文档文本，按一定长度切成 chunk，调用外部 Embedding API 生成向量，最后写入 LanceDB。到那一步之后，P7 才能在聊天时先检索知识库，再把命中的 chunk 作为 sources 返回。
