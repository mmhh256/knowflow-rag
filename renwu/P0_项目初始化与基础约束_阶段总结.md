## 二、我应该怎么运行和测试

第一次运行前，执行：

```bash
npm install
```

启动开发服务器：

```bash
npm run dev
```

浏览器访问：

```txt
http://localhost:3000
```

确认 P0 完成的方法：

1. 首页能正常打开，能看到 `Next.js Agentic RAG Knowledge Base`。
2. 页面样式正常，比如居中布局、卡片、文字颜色等 Tailwind 样式生效。
3. 项目根目录存在 `.env.example`。
4. 项目存在 `src/lib/config.ts`。
5. 项目存在基础目录：`src/components`、`src/lib`、`src/store`、`src/app/api`、`uploads`、`data/lancedb`。
6. 执行下面命令检查构建：

```bash
npm run build
```

也可以执行：

```bash
npm run lint
```

## 三、本阶段涉及的知识点

### Next.js App Router 是什么

App Router 是 Next.js 新一代路由系统。它用 `app` 目录里的文件和文件夹来决定页面地址。

比如当前项目使用 `src/app/page.tsx`，它对应网站首页 `/`。

`src/app/layout.tsx` 是根布局，所有页面都会被它包起来。以后如果创建 `src/app/chat/page.tsx`，就会得到 `/chat` 页面；如果创建 `src/app/api/chat/route.ts`，就会得到后端接口 `/api/chat`。

### TypeScript 在项目里的作用

TypeScript 是 JavaScript 的类型增强版。它可以提前检查变量、函数参数、返回值是否符合预期。

在这个项目里，TypeScript 能帮助我们减少很多低级错误。比如 `src/lib/config.ts` 中会明确配置对象有哪些字段，后续写代码时如果拼错字段名，编辑器和构建命令会更早提示。

### Tailwind CSS 是什么

Tailwind CSS 是一个工具类 CSS 框架。它不是先写很多 CSS 类名再去页面里引用，而是直接在 `className` 里组合工具类。

例如：

```tsx
<main className="flex min-h-screen items-center justify-center">
```

这表示页面使用 Flex 布局、最小高度为一屏，并且内容水平垂直居中。

### 为什么要提前规划目录结构

RAG 项目后续会有很多模块：聊天 UI、文档管理、模型设置、数据库、文档解析、向量检索、AI 调用、SSE 流式输出等。

如果一开始不规划目录，后面很容易把所有代码都堆在一个文件里，项目会很难维护。P0 先把目录边界划清楚，后续每个阶段就知道代码应该放在哪里。

### `.env.example` 是干什么的

`.env.example` 是环境变量模板。它告诉开发者这个项目需要哪些配置，但不会放真实密码或 API Key。

真正运行项目时，可以复制它生成 `.env.local`，然后在 `.env.local` 里填写自己的密钥。`.env.local` 不应该提交到 Git。

### 为什么 API Key 不能放前端

前端代码会被打包发送到浏览器，用户可以通过浏览器开发者工具看到很多前端资源。

如果把 API Key 写进 React 组件，或者用 `NEXT_PUBLIC_` 暴露密钥，别人就可能拿到你的密钥并调用你的模型服务，造成费用损失和安全风险。

正确做法是：前端只请求自己的 Next.js 后端接口，由后端读取环境变量里的 API Key，再由后端去调用外部模型 API。

### `lib/config.ts` 为什么要统一管理环境变量

如果每个文件都直接写 `process.env.xxx`，后续很容易出现变量名不统一、默认值不一致、类型转换混乱的问题。

统一放到 `src/lib/config.ts` 后，其他代码只需要读取同一个配置对象。这样更容易维护，也更容易在后续阶段做校验、默认值和安全控制。

## 四、本阶段的重难点

P0 阶段最容易出错的地方有：

- 目录结构混乱：一开始没有分清页面、组件、服务端逻辑、RAG 逻辑和数据目录，后续会很难扩展。
- 环境变量命名不统一：比如同一个含义一会儿叫 `LLM_API_KEY`，一会儿叫 `OPENAI_KEY`，后续调试会很痛苦。
- 把后端密钥暴露到前端：API Key 不能写在 React 组件里，也不要加 `NEXT_PUBLIC_`。
- 一开始就提前写后面阶段的功能：P0 只做工程基础，不做数据库、AI、登录、RAG、文档上传、SSE。
- 空目录无法被 Git 跟踪：所以本阶段使用 `.gitkeep` 保留目录结构。
- 忽略 Next.js 当前版本文档：本项目使用 Next.js 16，开发前需要优先看本地 `node_modules/next/dist/docs/` 的说明。

## 五、面试时我应该怎么说

在 P0 阶段，我主要完成了项目工程初始化和基础约束设计。这个项目采用 Next.js App Router、React、TypeScript 和 Tailwind CSS 作为基础技术栈。我先保证项目可以正常启动和访问首页，然后规划了后续 RAG 系统会用到的目录结构，比如聊天组件、文档管理组件、模型设置组件、RAG 逻辑、LLM 适配层、文档处理、向量库和状态管理目录。同时我创建了 `.env.example` 和 `src/lib/config.ts`，把环境变量的命名和读取方式提前统一起来，避免后续开发中出现配置分散、密钥误暴露的问题。这个阶段我没有提前接入数据库、AI、登录或 RAG 功能，而是先把工程边界和安全规范打好基础，方便后续按阶段稳定迭代。
